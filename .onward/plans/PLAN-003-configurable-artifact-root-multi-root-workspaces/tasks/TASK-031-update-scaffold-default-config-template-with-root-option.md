# Context

The default `.onward.config.yaml` generated by `onward init` should show the new `root` option as a commented-out example so users know it's available.

# Scope

- In `scaffold.py`, locate the `DEFAULT_FILES` entry for `.onward.config.yaml` (or the config template string).
- Add a commented-out `# root: .onward` line with a brief comment explaining the option.
- Optionally add a commented-out `# roots:` block showing multi-root syntax.
- Keep the default behavior unchanged — the config still works without uncommenting anything.

# Out of scope

- Parameterizing scaffold functions (TASK-015).
- Changing init behavior (TASK-016).
- Full documentation updates (TASK-030).

# Files to inspect

- `src/onward/scaffold.py` — `DEFAULT_FILES` dict or config template string

# Implementation notes

- The config template is a YAML string embedded in scaffold.py. Add the new lines in a logical position (near the top, after any existing config keys).
- Example addition:
  ```yaml
  # Artifact root directory (default: .onward)
  # root: .onward
  #
  # Multi-root workspaces:
  # roots:
  #   frontend: .fe-plans
  #   backend: .be-plans
  # default_project: frontend
  ```
- Keep it concise — the full docs are in INSTALLATION.md.

# Acceptance criteria

- `onward init` in a fresh directory generates a config file containing commented-out `root` option.
- The generated config is valid YAML (commented lines don't break parsing).
- Uncommenting `root: .onward` and re-running `onward init` still works.

- This is the smallest task in the plan. Can be done at any time since it only touches the template string.
